PHASE 1: Search Data Infrastructure & Firestore Indexing
Goal: Prepare your Firestore data and create a lightweight search index so queries are fast and cost-efficient.

CONTEXT:
You are working on PakCart — a Pakistani e-commerce React SPA (React 18 + TypeScript + Vite) 
that uses Cloud Firestore as its only database. There is NO traditional backend — all 
business logic runs in the browser via services in client/src/services/. The state management 
uses Zustand (auth + cart) and TanStack Query v5 (object-form only). Routing is via wouter. 
Styling is Tailwind CSS + shadcn/ui.

The product data model (from shared/schema.ts) includes: slug, name, price, profit, images, 
categoryId, variants (color/size with per-option price overrides), labels (Best Seller, New, etc.), 
rating, reviewCount, inStock, active.

Category model supports 2-level hierarchy: slug, name, parentCategoryId.

TASK — PHASE 1: Search Data Infrastructure

1. CREATE a new Firestore collection called "searchIndex" that acts as a lightweight, 
   denormalized search index. Each document should mirror a product but contain ONLY 
   the fields needed for search:
   - productId (string — reference to original product doc ID)
   - name (string — the product name)
   - nameLower (string — lowercase version of name for case-insensitive matching)
   - nameTokens (array of strings — every word in the name, lowercased, plus partial 
     prefixes of each word down to 2 characters. Example: "Leather Handbag" → 
     ["leather", "leat", "lea", "le", "handbag", "hand", "han", "ha", "handb", "handba"])
   - categoryName (string — the human-readable category name)
   - categorySlug (string)
   - parentCategoryName (string — if exists)
   - price (number)
   - primaryImage (string — first image URL)
   - slug (string — product slug for navigation)
   - labels (array — e.g., ["Best Seller", "New"])
   - inStock (boolean)
   - active (boolean)
   - rating (number)
   - reviewCount (number)
   - searchScore (number — computed as: (rating × reviewCount × 0.3) + 
     (labels.includes("Best Seller") ? 50 : 0) + (inStock ? 20 : 0))

2. CREATE a service file: client/src/services/searchIndexService.ts
   This service should have:
   - buildSearchIndex() — reads all products + categories, builds the denormalized 
     searchIndex documents, and writes them to Firestore in batches (batch writes, 
     max 500 per batch). This is an admin-only function.
   - updateSearchIndexForProduct(productId) — updates a single product's search 
     index entry. Call this whenever a product is created/updated/deleted from the 
     admin panel.
   - deleteSearchIndexForProduct(productId) — removes entry when product is deleted.
   - generateTokens(name: string): string[] — the tokenization function that creates 
     prefix tokens from a product name.

3. UPDATE the existing product CRUD in client/src/services/ so that every 
   addProduct, updateProduct, deleteProduct call ALSO calls the corresponding 
   searchIndex update function. This keeps the index in sync.

4. ADD these Firestore composite indexes to firestore.indexes.json:
   - Collection: searchIndex, fields: [nameTokens (array-contains), searchScore (desc)]
   - Collection: searchIndex, fields: [nameTokens (array-contains), price (asc)]
   - Collection: searchIndex, fields: [nameTokens (array-contains), price (desc)]
   - Collection: searchIndex, fields: [categorySlug (asc), searchScore (desc)]
   - Collection: searchIndex, fields: [active (asc), inStock (asc), searchScore (desc)]

5. ADD an admin button in the admin panel (any convenient location) labeled 
   "Rebuild Search Index" that calls buildSearchIndex() with a loading state and 
   success/error toast.

TASK — PHASE 2: Core Search Service & Query Logic

1. CREATE client/src/services/searchService.ts with these functions:

   a) searchProducts(query: string, options?: SearchOptions): Promise<SearchResult[]>
      - SearchOptions type: { limit?: number (default 10), sortBy?: 'relevance' | 'price_asc' | 
        'price_desc' | 'rating' | 'newest', categorySlug?: string, minPrice?: number, 
        maxPrice?: number, inStockOnly?: boolean }
      - SearchResult type: { productId, name, slug, price, primaryImage, categoryName, 
        categorySlug, rating, reviewCount, labels, inStock, relevanceScore }
      - Logic:
        → Lowercase and trim the query
        → Split into individual words
        → For single-word queries: use Firestore array-contains on nameTokens with that word
        → For multi-word queries: use array-contains for the LONGEST word (Firestore 
          limitation — only one array-contains per query), then client-side filter results 
          to ensure ALL other words also appear in nameTokens
        → Apply sortBy ordering via Firestore orderBy when possible, else sort client-side
        → Apply price range filtering client-side (Firestore can't do range on price AND 
          array-contains on tokens simultaneously)
        → Filter out active === false products always
        → If inStockOnly, filter out inStock === false
        → Calculate relevanceScore client-side: base is document's searchScore, add bonus 
          points for exact name match (+100), name starts with query (+50), and number of 
          token matches (+10 per match)
        → Return sorted by relevanceScore desc (if sortBy is 'relevance')

   b) getSuggestions(query: string): Promise<Suggestion[]>
      - Returns max 7 suggestions
      - Suggestion type: { text: string, type: 'product' | 'category' | 'query', 
        slug?: string, image?: string }
      - Logic:
        → If query length < 2, return empty array
        → Run a Firestore query on searchIndex where nameTokens array-contains 
          the lowercased query, ordered by searchScore desc, limit 5
        → Separately, search categories collection where nameLower starts with query 
          (use >= and < range query trick for prefix matching), limit 2
        → Product results become type: 'product' suggestions (include image + slug)
        → Category results become type: 'category' suggestions (include slug)
        → Merge and return

   c) getPopularSearches(): Promise<string[]>
      - Query searchIndex ordered by searchScore desc, limit 8
      - Extract unique category names from results
      - Return as popular search terms (e.g., ["Women's Bags", "Men's Watches", ...])
      - Cache this result in TanStack Query with staleTime of 10 minutes

   d) getRecentSearches(): string[]
      - Read from localStorage key "pakCart_recentSearches"
      - Return array of last 5 searches
      
   e) saveRecentSearch(query: string): void
      - Save to localStorage, max 5 entries, no duplicates (move to front if exists)
      - Trim and ignore empty strings
