TIER 2: HIGH-IMPACT FIXES (Performance band 40–60)
Prompt 4: Font Loading & CLS Prevention
CopyAudit all font usage:

1. Which fonts are critical for FCP? (only headlines/branding) vs. decorative?
2. For each critical font:
   - Is it self-hosted or CDN? (self-hosted + preload is faster)
   - Are variable font files being used (smaller than multiple weights)?
   - Is font-display: swap applied? (allows fallback text to render immediately)
   - Is font-size-adjust being used to reduce layout shift when font loads?

3. For decorative fonts:
   - Can they be loaded deferred (font-display: optional)?
   - Are they being preloaded unnecessarily?

Then:
- Measure layout shift caused by font swapping: use Lighthouse CLS metric before/after each change
- Consider using system fonts for body text (faster, familiar to users) and only load branded fonts for h1/logo
- Generate a font subsetting strategy: do you need all Unicode glyphs or just Latin + numbers?

Create a report: "Estimated CLS improvement: X → 0.0Y after font loading fixes"

Prompt 8: Interaction to Next Paint (INP) & Input Responsiveness
CopyINP is Google's new responsiveness metric (replacing FID in 2024).

Test on mobile device or Chrome DevTools emulation:
1. Homepage: click a category card, measure delay before next paint
2. Product page: click "Add to Cart", measure button feedback delay
3. Category page: type in search box, measure filter update latency

For each slow interaction (> 200 ms):
1. Use Chrome DevTools Performance tab → record interaction → find what's blocking the main thread
2. Questions to ask:
   - Is a large function running synchronously?
   - Is React re-rendering too much?
   - Is a network request blocking the UI?
3. Solutions:
   - Move work to Web Worker or requestIdleCallback()
   - Batch updates with useTransition() (React 18+)
   - Pre-fetch data before interaction (eager loading)
   - Debounce expensive operations (e.g., search filter)

Report: "INP baseline: X ms → target: < 200 ms after Y fixes"

TIER 5: BUSINESS LOGIC OPTIMIZATIONS (Specific to PakCart)
Prompt 11: Product Grid & Category Filtering Performance
CopyOn category collection pages, when user applies filters (price, size, color, etc.):

Current state: Likely using TanStack Query to fetch filtered products from Firestore.

Optimization opportunities:
1. **Client-side filtering**: If the full product list is < 500 items, cache on client and filter locally
   - Reduces network latency from 200–500 ms to 0–50 ms
   - Trade-off: initial load is slightly larger but subsequent filters are instant

2. **Pagination / Virtual scrolling**: 
   - Is the page loading all 100 products at once? Paginate or use react-window for virtual scrolling
   - Benefit: fewer DOM nodes → faster re-renders

3. **Search debouncing**: 
   - Does search trigger Firestore query on every keystroke? Debounce to 300–500 ms
   - Saves bandwidth and latency

4. **Aggregate data**: 
   - Are you fetching min/max price every time? Cache in Firestore doc or Redis
   - Benefit: filter dropdowns load instantly

Create a performance profile of the category page:
- Time to fetch product list
- Time to render grid (depends on React re-renders)
- Time to apply first filter
- After fixes, measure again and document improvement

CopyProduct detail & checkout pages often have high abandonment due to slow load:

   - If loading hundreds of reviews, paginate or load on-demand

4. **Checkout form**:

5. **Payment gateway**:
